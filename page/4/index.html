Using Windows 7 to host PHP applications in 5 easy steps!

A few people have asked me recently whether it’s possible to setup Windows 7 as a PHP server (for development purposes). The answer is absolutely yes, and it’s a breeze to setup. Follow these five simple steps to get PHP up and running in minutes:

  1. In the Programs and Features control panel, click on the Turn Windows features on or off link:

  1. In the list of Windows Features, expand Internet Information Services, World Wide Web Services, and the Application Deployment Features. If it’s not already, select the CGI checkbox and click OK. (The most reliable way of hosting PHP applications on Windows 7 is to use the built in FastCGI interface for IIS – checking this box installs it together with any pre-requisites.)

  1. Download the non-thread-safe (NTS) version of PHP from http://www.php.net/downloads.php. The current version (as of time of writing is 5.2.9). (The thread safe (TS) version will also work, but generally NTS is faster, and thread safety is not an issue under FastCGI). Expand the zip to an installation directory of your choice – e.g. c:devphp

  2. Copy the php.ini-recommended file to php.ini in the PHP directory. Edit the php.ini file and add correctly configure extension_dir, pointing to the PHP extensions directory (normally the .ext folder of the PHP installation – e.g c:devphpext). You can also configure other php.ini options and modules here if required.

  3. Run Internet Information Services Manager by typing inetmgr in the Start menu. You can either set the global settings of the server, or (recommended) add a new web site to run the PHP applications. Once you’ve done this, double click on the Handler Mappings for the site and add a new module mapping with the following settings:

Request path should be set to .php. Module should be FastCgiModule. Executable should be {php_install_dir}php-cgi.exe. *Name can be anything – I use “PHP via Fast CGI”.

That’s it! Simply start/restart IIS and you are ready to go. The easiest way to test that everything is working is to create a simple info.php file with a single line: <div class="code">

&lt;?php phpinfo(); ?&gt;

</div> When you access this page from a browser (e.g. http://localhost:8081/info.php), you should see the PHP info screen:

Validate that the server API is using CGI/FastCGI and that the loaded configuration file is the one in your installation directory.

User Interface (UI) vs. User Experience (UX)

I often get asked the difference between User Interface (UI) and User Experience (UX). Although the original intent and definition of UI started out in the right direction, too many times I've heard it used to explain just the visual aspects (or cool factor) of an interface.&nbsp; To give you a non-technology related example, let me show you the hotel room that I recently stayed at in Dublin.&nbsp;&nbsp;The hotel is&nbsp;one of the more boutique, modern style ones that we see in large cities (imagine Morcheeba playing in the lobby, contrasting black and white furniture&nbsp;etc. - you get the idea).

As I got to my room late at night, I of course wanted to switch on the lights.&nbsp; In front of me (in semi-darkness)&nbsp;I found&nbsp;the panel for the lights:

As you can see, on first glance, a relatively nice&nbsp;looking "interface" (stainless steel finish, clean, elegant etc.).&nbsp;&nbsp;The&nbsp;experience however was a different story.&nbsp; Firstly, there&nbsp;is no indication&nbsp;of what each switch does.&nbsp; After playing around for a minute, I was able to work out that&nbsp;the left dial is for the main lights, the right dial is for the bathroom lights, the main switch seems to be like a master switch for both.&nbsp; Seems simple enough, however you can also push the dials to turn them on/off individually, with no feedback or indication of what state they are in.&nbsp; To add more confusion, there is a 3 second delay after pushing the dial before the lights come on.&nbsp; Of course, what happens is the following:&nbsp; Hit the master switch - nothing happens.&nbsp; Turn the dial - nothing happens.&nbsp; Push the dial, nothing happens.&nbsp; Push the dial again (within 3 seconds of pushing the dial last time).&nbsp; Nothing happens.&nbsp; Hit the master switch again - nothing happens.&nbsp; Sigh.&nbsp; Repeat.

Once I did get the lights working in the bathroom,&nbsp;I came across the following sink:

Again, very nice looking - a&nbsp;modern, open design with elegant lines and faucets.&nbsp;&nbsp; Question for you: How do I release the water in the bowl?&nbsp; Looking around the basin, there is no stopper for the plug, no chain, no indication of how to release the plug.&nbsp; Also, notice how the plug is flat to the surface of the bowl - there is no affordance&nbsp;indicating that it might be possible to push or pull the plug in any way to release the water.&nbsp; The answer?

Ah, of course - who wouldn't have thought about tipping the plug 45' to the left and having it swivel in the middle?&nbsp; Sigh (again).&nbsp; Having been humiliated with the sink, I thought I would take a shower instead.&nbsp; I was greeted with this shower head.&nbsp;

Again, the pencil design looks very elegant, and unique - but just try moving the direction of the shower head when the water is running.&nbsp; Impossible.&nbsp; Because of the slim design and texture, you can't redirect the shower head in any direction (which can be annoying if it's aiming toward the door as you get in).&nbsp; Also, the design results in a powerful, but very directed spray of water - certainly refreshing, but not as comfortable as you may like.

The difference between&nbsp;"cool&nbsp;looking UI"&nbsp;and UX?&nbsp; You could argue that the items in the hotel room look as if they have a great design, but ultimately&nbsp;result in&nbsp;a poor user experience.&nbsp;I like unique&nbsp;designs - but don't let this come at the expense of a frustrating experience.&nbsp; I just hope we don't see the same thing with new UI technologies.&nbsp; There are a lot of new UI advances that can create&nbsp;unique looking interfaces, but without an understanding of true user experience, it's going to be as bad as my hotel room…

Building Browser Helper Objects using Managed Code

<font face="Verdana">Over the past&nbsp;few days I've been&nbsp;exploring the&nbsp;extensibility of IE7 from a user experience perspective.&nbsp; One of the options for creating an Add-On for IE is something called the Browser Helper Object (BHO for short).&nbsp; With&nbsp;a BHO you can trap a variety of events in Internet Explorer (such as browsing for a page, or when the user quits the browser), examine the results and take actions as required.&nbsp; This powerful feature allows you to build UIs that really integrate with the browsing experience.</font>

<font face="Verdana">How do I write one in .NET?</font>

<font face="Verdana">BHOs sound great, but the question is - how do I write one in .NET?&nbsp; Although the BHO interface is COM-based, it's not as hard as it sounds.&nbsp; Firstly, you must create a class library in .NET and add the following COM references:</font>

<font face="Verdana"></font><font face="Courier"> [ComImport(), Guid("fc4801a3-2ba9-11cf-a229-00aa003d7352")] [InterfaceType(ComInterfaceType.InterfaceIsIUnknown)] public interface IObjectWithSite { void SetSite([In ,MarshalAs(UnmanagedType.IUnknown)] object pUnkSite); void GetSite(ref Guid riid, [MarshalAs(UnmanagedType.IUnknown)] out object ppvSite); }</font>

<font face="Verdana">This interface provides the GetSite and SetSite for hooking into events within IE.&nbsp; After you have these defined, create a class that overrides IObjectWithSite:</font>

<font face="Courier">[ClassInterfaceAttribute(ClassInterfaceType.None)] [GuidAttribute(MyGuid)] [ProgIdAttribute("MyNamespace.MyBHO")] public class MyBHO: IObserver, IObjectWithSite {</font> <p><font face="Courier">}</font> <p><font face="Verdana">In this class you'll need to define a browser class object and browser events object from the Internet Controls library (VS.NET will create an Interop assembly called Interop.SHDocVw when you import this):</font>

<font face="Courier">protected SHDocVw.IWebBrowser2 m_pIWebBrowser2; protected SHDocVw.DWebBrowserEvents2_Event m_pDWebBrowserEvents2; </font> <p><font face="Verdana">With these defined, you now override GetSite and SetSite, initializing the Internet Controls and enabling the events:</font>

<font face="Courier">public void SetSite(object pUnkSite) { </font><font face="Courier"> if (m_pIWebBrowser2!=null) Release(); </font> <p><font face="Courier">if (pUnkSite==null) return; </font> <p><font face="Courier">m_pIWebBrowser2 = pUnkSite as SHDocVw.IWebBrowser2;&nbsp; // set the reference</font>

<font face="Courier">if (!(m_pIWebBrowser2.FullName.ToUpper().EndsWith("IEXPLORE.EXE")))&nbsp; //make sure only runs on IE { Release(); return; } </font>

<font face="Courier">m_pDWebBrowserEvents2 = m_pIWebBrowser2 as SHDocVw.DWebBrowserEvents2_Event ; </font> <p><font face="Courier">if (m_pDWebBrowserEvents2 != null) { m_pDWebBrowserEvents2.BeforeNavigate2 += new SHDocVw.DWebBrowserEvents2_BeforeNavigate2EventHandler(BeforeNavigate); m_pDWebBrowserEvents2.OnQuit += new SHDocVw.DWebBrowserEvents2_OnQuitEventHandler(OnQuit); } else {

Release(); return; } }</font> <p><font face="Courier"></font>&nbsp; <p><font face="Courier">public void GetSite(ref System.Guid riid, out object ppvSite) { ppvSite=null; if (m_pIWebBrowser2 != null) { IntPtr pSite = IntPtr.Zero; IntPtr pUnk = Marshal.GetIUnknownForObject(m_pIWebBrowser2); Marshal.QueryInterface(pUnk, ref riid, out pSite);

Marshal.Release(pUnk); if (!pSite.Equals(IntPtr.Zero)) { ppvSite = pSite; } else { Release(); Marshal.ThrowExceptionForHR(E_NOINTERFACE); } } else { Release(); Marshal.ThrowExceptionForHR(E_FAIL); } }</font> <p><font face="Verdana">As you can see in the SetSite method, you can now define the hooks within Internet Explorer:</font> <font face="Courier">m_pDWebBrowserEvents2.BeforeNavigate2 += new SHDocVw.DWebBrowserEvents2_BeforeNavigate2EventHandler(BeforeNavigate); m_pDWebBrowserEvents2.OnQuit += new SHDocVw.DWebBrowserEvents2_OnQuitEventHandler(OnQuit); </font>

<font face="Verdana"><font face="Verdana">These can&nbsp;point to&nbsp;your own methods that can act on these events accordingly (e.g. launch a particular process with a given URL, rewrite URLs etc.)</font></font>

<font face="Verdana">How to install a BHO using an MSI</font>

<font face="Verdana">One of the challenges is not building a BHO, but getting it to install/uninstall reliably through an MSI.&nbsp; To run correctly,&nbsp;a BHO requires registration via COM interop, but setting the vsdrpCOM property in the MSI isn't enough (this results in a successful registration most of the time, but the add-in won't remove itself at uninstall).</font>

<font face="Verdana">My recommendation is to set the MSI property to vsdrpNoDotRegister and instead create your own registration component.&nbsp; This can be a simple command prompt executable that you run as a custom action (for both install and uninstall) in the MSI.&nbsp; </font>

<font face="Verdana">To register the assembly, use the following code within your registration code:</font>

<font face="Verdana"><font face="Courier">System.Runtime.InteropServices.RegistrationServices r = new System.Runtime.InteropServices.RegistrationServices(); r.RegisterAssembly(System.Reflection.Assembly.LoadFile(bhoDll), System.Runtime.InteropServices.AssemblyRegistrationFlags.SetCodeBase);</font> </font>

<font face="Verdana">To unregister the assembly, you use the UnregisterAssembly method:</font>

<font face="Courier">System.Runtime.InteropServices.RegistrationServices r = new System.Runtime.InteropServices.RegistrationServices(); r.UnregisterAssembly(System.Reflection.Assembly.LoadFile(bhoDll)); </font>

<font face="Verdana">UAC Considerations for Vista</font>

<font face="Verdana">This approach works all very well for XP, but if you are installing your BHO on Vista you'll need to set the user access control correctly.&nbsp; One of the problems is that custom actions in MSIs run as an non-elevated user, which means that unless you are running the MSI as elevated your custom action won't have the permissions required to register the BHO.&nbsp; I haven't found a way to elevate a user manually from a piece of managed code, but you can overcome this using one of two ways:</font>

<font face="Verdana">1.&nbsp; Write a batch file that contains the following:</font>

<font face="Courier">msiexec /i MyMSI.msi</font>

<font face="Verdana">…and right click on the batch file, and select "Run as Administrator"</font>

<font face="Verdana">2.&nbsp; Use the ORCA tool (part of the Windows SDK) to set the custom actions to run as SYSTEM.</font>

<font face="Verdana">To do this, download the SDK, install and run ORCA and open your MSI.&nbsp; Open the custom action from the left hand pane, select your custom action and set the type to 3090.&nbsp; This will cause the custom action to run as the SYSTEM account and registration will be successful.</font>

<font face="Verdana">Examples</font>

<font face="Verdana">Finally, if you are looking for a great example of a managed BHO (and more details of how one works), check out this post from Michele Leroux Bustamante.&nbsp; Here, Michele shows how to use a BHO for observing user behavior in IE, with downloadable code that you can build your own from.</font>
